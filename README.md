# Поток пациентов v2 — Clinic Cards

Канбан-борд потока пациентов стоматологической клиники. UI реализован 1:1 по
дизайну из Claude Design; данные о пациентах тянутся **из Clinic Cards API**
(только чтение), а позиции карточек на борде хранятся в **своей БД**
(libSQL/Turso — SQLite-совместимая, локально файл, в проде serverless).

```
Браузер ──/api──► Express (server/) ──Token──► Clinic Cards API   (read-only, GET)
                       │
                       └── libSQL/Turso   ← позиции карточек + кэш выгрузки CC
```

Разворачивается двумя способами:
- **Один Node-хост** (локально, Railway, Render): `npm run build && npm start` —
  Express отдаёт `dist/` и API с одного порта.
- **Vercel** (serverless): фронт — статика на CDN, бэкенд — функция `api/index.js`,
  БД — Turso. См. [«Деплой на Vercel»](#деплой-на-vercel).

## Быстрый старт

```bash
npm install
cp .env.example .env         # если .env ещё нет
# вставьте CLINIC_CARDS_API_KEY в .env  (можно оставить пустым — будет ДЕМО-режим)
npm run dev                  # backend :8787 + frontend :5173 одновременно
# откройте http://localhost:5173
```

- `npm run dev` — бэкенд и фронтенд вместе (concurrently).
- `npm run dev:web` / `npm run dev:api` — по отдельности.
- `npm run build && npm start` — прод-сборка; Express отдаёт `dist/` и API с одного порта (`:8787`).

## .env

| Переменная | Назначение |
| --- | --- |
| `CLINIC_CARDS_API_KEY` | Ключ Clinic Cards. **Пусто → демо-режим на мок-данных.** |
| `CLINIC_CARDS_BASE_URL` | База API (по умолчанию `https://cliniccards.com/api`). |
| `CLINIC_CARDS_CLOSED_STATUSES` | Названия статусов CC = «закрыт/архив» — такие пациенты скрыты. Для этой клиники: `Выполненный,Отказался`. |
| `CLINIC_CARDS_PLANS_SINCE` | С какой даты искать планы лечения для поля «Послуга». |
| `PORT` | Порт бэкенда (8787). |
| `CC_CACHE_TTL_SECONDS` | TTL кэша выгрузки из CC (лимит API — 60 запросов/мин). |
| `TURSO_DATABASE_URL` | libSQL/Turso URL. **Пусто → локальный файл** `./data/board.db`. Для прода — `libsql://…`. |
| `TURSO_AUTH_TOKEN` | Токен Turso (для прод-БД). |
| `DB_PATH` | Путь к локальному файлу БД (когда `TURSO_DATABASE_URL` пуст). |
| `WINDOW_DAYS` | Сколько дней показывать пациента после создания заявки; переход на этап продлевает окно (по умолч. 30). |
| `STUCK_DAYS` | Без перехода столько дней → «Потребують уваги» (3). |
| `HOT_VISIT_DAYS` | «Гарячі»: ближайший визит в пределах N дней (1). |
| `CONVERSION_WINDOW_DAYS` | Окно расчёта конверсии (30). |

> Ключ живёт только на бэкенде и никогда не попадает в браузер. `.env` и
> `data/` — в `.gitignore`.

## Деплой на Vercel

Vercel — serverless, поэтому файловой SQLite и фонового процесса там нет. Данные
хранятся в **Turso** (libSQL, SQLite-совместимая), а Express обёрнут в функцию
`api/index.js`. Clinic Cards дёргается не чаще раза в `CC_CACHE_TTL_SECONDS`
(выгрузка кэшируется в Turso и переживает холодные старты).

1. **Turso БД:**
   ```bash
   curl -sSfL https://get.tur.so/install.sh | bash   # или brew install tursodatabase/tap/turso
   turso auth login
   turso db create board
   turso db show board --url            # → TURSO_DATABASE_URL
   turso db tokens create board         # → TURSO_AUTH_TOKEN
   ```
2. **Импорт в Vercel:** New Project → выберите этот GitHub-репозиторий. Framework
   Preset — *Other* (сборка и роутинг заданы в `vercel.json`).
3. **Environment Variables** (Project → Settings → Environment Variables):
   - `CLINIC_CARDS_API_KEY` — ваш ключ Clinic Cards
   - `TURSO_DATABASE_URL`, `TURSO_AUTH_TOKEN` — из шага 1
   - при желании: `CLINIC_CARDS_CLOSED_STATUSES`, `CC_CACHE_TTL_SECONDS`,
     `WINDOW_DAYS`, `STUCK_DAYS`, `HOT_VISIT_DAYS`, `VISITS_BACK_DAYS`/`VISITS_FWD_DAYS`
4. **Deploy.** Vercel соберёт фронт (`npm run build` → `dist`, статика на CDN) и
   поднимет `api/index.js`. Фронтенд ходит на `/api/*` того же домена.

Локально тот же код работает на файловой БД (`TURSO_DATABASE_URL` пустой) —
менять ничего не нужно. Альтернатива Vercel — один Node-хост (Railway/Render):
подключите репозиторий, `npm run build && npm start`, задайте те же env
(Turso не обязателен — можно оставить локальный файл на диске хоста).

## Как это работает

- **Язык интерфейса — украинский.** Вся клиентская часть на украинском.
- **Воронка (7 этапов).** `Консультація призначена` → `Прийшов на консультацію`
  → `Направлений на КТ` → `План лікування складено` → `Лікування в процесі` →
  `Лікування завершено`, плюс ветка `Втрачений / Відмова`. Новые пациенты падают
  в первую колонку **«Консультація призначена»**.
- **Источник пациентов.** Бэкенд тянет `GET /patients` (с пагинацией),
  `GET /patientStatuses`, `GET /staff`, `GET /plans` и собирает карточки.
  Clinic Cards **не изменяется** — только чтение.
- **Кто на борде.** Пациенты со статусом из `CLINIC_CARDS_CLOSED_STATUSES`
  скрыты. Плюс **окно показа**: пациент виден `WINDOW_DAYS` дней от создания
  заявки; каждый переход на следующий этап продлевает окно ещё на `WINDOW_DAYS`
  от даты перехода. Просроченные окном — сходят с борда (у клиники это ужимает
  ~2000 старых заявок до тех, кто реально в работе).
- **«Потребують уваги».** Если пациент не двигался `STUCK_DAYS` дней — вкладка
  «Потребують уваги» рядом с «Гарячі»; на карточке — бейдж «ЗАСТРЯГ · N дн» и
  между какими этапами застрял (`поточний → наступний`).
- **«Гарячі».** Пациенты с ближайшим визитом в пределах `HOT_VISIT_DAYS`.
- **Напоминание «відвідав клініку — перемістити».** Если у пациента есть визит
  (`/visits`), время которого **прошло уже после того, как он вошёл в текущий
  этап** (`visit_start > entered_at`), и он в активном этапе — карточка
  подсвечивается синим с закрываемой пометкой «Був у клініці · перемістити» и
  датой визита. Отслеживание без хранения истории: сравниваем время визита из
  свежей выгрузки CC с нашим `entered_at`. Закрытие пометки пишет маркер
  `reminder_dismissed_at` (время того визита) — **новый** визит позже этой даты
  снова поднимет напоминание. Такие карточки тоже попадают во вкладку
  «Потребують уваги».
- **Позиции — наши.** Когда карточку двигают (кнопка «Переместить» в панели
  пациента), новый этап сохраняется в SQLite (`positions`). При следующей
  выгрузке пациент показывается уже в сохранённой колонке.
- **SLA** считается от момента входа в текущий этап (`entered_at`) против
  норматива этапа; при перемещении отсчёт сбрасывается. Для новых пациентов
  `entered_at` = момент импорта (а не дата создания в CC), поэтому старые
  карточки не влетают сразу «просроченными».
- **Обновление.** При загрузке + кнопка обновления в шапке + автополлинг раз в
  5 минут (`POLL_MS`).

### Производительность и нагрузка на API (лимит CC — 60 запросов/мин)

- **Кэш выгрузки CC** в памяти (`CC_CACHE_TTL_SECONDS`, по умолч. 180 с).
- **Stale-while-revalidate:** после первого прогрева запросы никогда не ждут
  Clinic Cards — отдаётся кэш, а обновление идёт в фоне. Значит CC дёргается
  максимум раз в TTL, **независимо от числа клиентов и частоты поллинга**.
- **Мемоизация собранного борда:** пересборка только при изменении выгрузки или
  позиции карточки; иначе ответ ~10 мс без обращения к БД.
- **gzip** на все ответы (борд на 2000+ пациентов ужимается в ~5 раз).
- **Окно рендера в колонках:** первая колонка может держать тысячи карточек
  (все новые лиды падают туда). Рендерятся ~25 карточек, остальные догружаются
  при скролле / по кнопке «Показать ещё» — счётчик колонки показывает полное
  число. Первая отрисовка ~300 мс вместо ~3 с.

### Конверсия (реальная, вместо заглушки 34%)

Каждый переход по этапам логируется (таблица `transitions`) с признаком
«вовремя» = уложился в норматив (SLA) этапа-источника. В «Пульсе дня»:
- **`X% вчасно по етапах`** — доля переходов в срок за `CONVERSION_WINDOW_DAYS`.
- **`Консультація → Прийшов: N (Y%)`** — переходы «Консультація призначена» →
  «Прийшов на консультацію». Пока переходов нет — «поки немає переходів».

### Фильтры

- **Поиск** по имени и телефону (в шапке).
- **Вкладки** «Всі заявки / Мої / Гарячі / Потребують уваги» (с счётчиками).
- **По куратору** — компактный селект «Куратор ▾» в «Пульсе дня» со списком
  кураторов и числами; выбор фильтрует борд, «Всі куратори» сбрасывает.
- **Окно рендера в колонках** — рендерится ~25 карточек, остальные догружаются
  скроллом / кнопкой «Показати ще» (счётчик колонки — полный).

### Маппинг полей CC → карточка

| Карточка | Clinic Cards |
| --- | --- |
| Имя / телефон | `firstname + lastname` / `phone` (RU `+7` и UA `+380`) |
| Коментар (осн. строка) | комментарий визита `GET /visits` → `note` (ближайший визит, иначе последний); фолбэк — название плана `plan_name` |
| Лікар (фильтр/аватар) | куратор (`curator` / `curator_id` → `staff`) |
| Відповідальний (аватар) | `curator` / `curator_id` → `staff` |
| Нотатка | `important_note`, иначе `Джерело: {source}` |
| Візит | `next_visit_date` (метки «Сьогодні/Завтра») |
| «Гарячий» | ближайший визит в пределах `HOT_VISIT_DAYS` (1) |

## API бэкенда

| Метод | Путь | Действие |
| --- | --- | --- |
| GET | `/api/board` | Борд: пациенты + админы + лента. `?refresh=1` — форс-выгрузка из CC. |
| POST | `/api/patients/:id/stage` | Переместить пациента (`{ "stage": "contact" }`), сохранить в БД. |
| POST | `/api/patients/:id/dismiss-followup` | Закрыть напоминание «відвідав клініку» (`{ "visitAt": "ISO" }`). |
| POST | `/api/refresh` | Форс-выгрузка из Clinic Cards. |
| GET | `/api/health` | Статус + режим (`live`/`mock`). |

## Структура

```
api/
  index.js        Vercel serverless entry (экспорт Express app)
server/
  config.js       env + определение этапов воронки
  clinicCards.js  read-only клиент Clinic Cards API (GET)
  db.js           libSQL/Turso: positions, transitions, cache
  mapper.js       CC-пациент → карточка борда (SLA, окно, застрял, визит)
  mockData.js     демо-данные для режима без ключа
  store.js        кэш выгрузки CC (память + Turso) + сборка борда
  app.js          Express-приложение (роуты) — без listen
  index.js        listen() для локали / Node-хоста
vercel.json       сборка фронта + роут /api/* → функция
src/
  api.js          клиент нашего бэкенда
  logic.js        computeView — вью-модель борда
  data.js         этапы/воронка (STAGES, CHAIN)
  App.jsx         состояние, поллинг, перемещения
  components/     Header, PulseBar, Board, PatientPanel, CrmFeed
```

## Ограничения (осознанные)

- Перемещение — через кнопку «Переместить» в панели (по воронке вперёд) и
  «Вернуть в работу» из веток «Думает»/«Потерян», как в дизайне. Drag-and-drop
  и произвольный выбор колонки можно добавить отдельно.
- Вкладка «Мои» без авторизации фильтрует по первому администратору
  (плейсхолдер до появления входа пользователей).
